# Contextual Retrieval — Anthropic's RAG upgrade

## The puzzle

You built RAG. You chunked your docs into 500-token blocks. You embedded them. Top-K retrieval works most of the time. But ~20% of queries return chunks that look right but lack the context to actually answer.

The chunk says: "It was reduced 15% YoY." Which thing was reduced? Which year? What entity? The chunk lost its context when you split the document.

**Contextual Retrieval** is Anthropic's fix: add per-chunk context generated by Claude before embedding. The result: 40–67% fewer retrieval failures, almost for free.

## The simple version

For each chunk you store:

1. Generate a 1–2 sentence context describing **what this chunk is about, what document it's from, and what came before**.
2. Prepend that context to the chunk before embedding.
3. Also use that context when computing BM25/keyword scores.

So instead of embedding:

```
It was reduced 15% YoY.
```

You embed:

```
This chunk is from ACME Inc's Q4 2024 earnings report. It describes operating expenses in the technology division.

It was reduced 15% YoY.
```

Now both the embedding and the keyword index know the chunk is about ACME, Q4 2024, and operating expenses. Retrieval becomes dramatically more accurate.

## The technical version

### Why standard RAG fails

Chunking destroys context. A 5,000-word document split into 10 chunks loses cross-references:

- Chunk 4 talks about "the second initiative" — but the list of initiatives is in chunk 2.
- Chunk 7 says "compared to last year" — but the year is named in chunk 1.
- Chunk 9 says "in this case" — but the case is set up in chunk 5.

A query like "what was the second initiative?" might not match chunk 4 at all (no word "initiative" appears there), even though chunk 4 has the answer.

Embeddings can't repair what was thrown away when the document was split.

### The Contextual Retrieval recipe

For each chunk, generate context with Claude:

```python
CONTEXTUAL_PROMPT = """
<document>
{whole_document}
</document>

Here is the chunk we want to situate within the whole document:

<chunk>
{chunk_content}
</chunk>

Please give a short succinct context to situate this chunk within the overall document for the purposes of improving search retrieval of the chunk. Answer only with the succinct context and nothing else.
"""

def generate_context(whole_doc: str, chunk: str) -> str:
    response = client.messages.create(
        model="claude-haiku-4-5",
        max_tokens=200,
        messages=[{
            "role": "user",
            "content": CONTEXTUAL_PROMPT.format(
                whole_document=whole_doc,
                chunk_content=chunk
            )
        }]
    )
    return response.content[0].text.strip()
```

Use Haiku for this — it's fast and cheap, and the task is well within its capability.

Then prepend the context to the chunk before embedding:

```python
augmented_chunk = f"{context}\n\n{chunk}"
embedding = embed(augmented_chunk)
bm25_text = augmented_chunk  # also index for keyword search
```

Store original chunk + augmented chunk + embedding + BM25 entry.

### Costs and how to manage them

Naively, generating context for every chunk = one Haiku call per chunk. For a 1M-token corpus chunked into 2,000 pieces, that's 2,000 calls.

**Prompt caching cuts this dramatically.** When you generate context for chunk N+1 in the same document, you re-send the whole document. Cache it, and each subsequent call costs only the chunk + output, not the whole document.

```python
client.messages.create(
    model="claude-haiku-4-5",
    max_tokens=200,
    system=[
        { "type": "text", "text": whole_document, "cache_control": { "type": "ephemeral" } }
    ],
    messages=[{ "role": "user", "content": f"Situate this chunk: {chunk}" }]
)
```

Anthropic's writeup shows context generation for a 1M-token corpus costing ~$1.02 (with caching), one-time. That's it. Then you embed once, store once, query forever.

### Hybrid: embeddings + BM25

The full Contextual Retrieval pattern from Anthropic's research:

1. **Contextual embeddings** — embed augmented chunks.
2. **Contextual BM25** — also index augmented chunks in a keyword index.
3. **At query time** — retrieve top-K from both, fuse the results (e.g. reciprocal rank fusion).
4. **Re-rank** — optionally use a re-ranker model on the fused set to pick final top-N.

Each component contributes:

- Contextual embeddings catch semantic matches.
- Contextual BM25 catches exact-term matches the embeddings miss.
- Re-ranking picks the best of the fused set.

Anthropic's reported numbers: contextual embeddings alone cut failures 35%; adding BM25 gets to 49%; adding re-ranking gets to 67% failure reduction vs. baseline RAG. Each layer adds work but compounds the gain.

### When Contextual Retrieval pays off

It pays off when:

- Your corpus has cross-references between chunks (most docs do).
- Chunks are short relative to the document (typical 500-token chunks).
- You're getting retrieval failures that look like "the chunk was right but didn't have enough context."
- You can afford the one-time generation cost (usually negligible with caching).

It's less needed when:

- Each chunk is self-contained (e.g. FAQ entries, product descriptions).
- The corpus is small enough to stuff (in which case skip RAG entirely).

### Implementation order

1. Get baseline RAG working without context augmentation.
2. Run an eval (Module 5.3) to measure retrieval failure rate.
3. Add context generation for each chunk, embed augmented chunks.
4. Re-run eval. Quantify the improvement.
5. If failures persist on edge cases, add BM25 hybrid.
6. If still room to improve, add re-ranking.

Don't build all four layers up front. Each adds complexity. Add the next layer only when the previous one isn't enough.

### Real eval numbers (Anthropic's writeup)

From Anthropic's published results on their internal test set:

- **Baseline RAG** (embeddings only, no context): ~5.7% failure rate at top-20.
- **Contextual Embeddings**: 3.7% failure rate (35% relative reduction).
- **Contextual Embeddings + Contextual BM25**: 2.9% failure rate (49% reduction).
- **+ Re-ranking**: 1.9% failure rate (67% reduction).

These numbers vary by corpus. The important thing isn't the absolute number — it's that each layer compounds, and the implementation cost is modest.

### Generating context — practical tips

- Use Haiku (cheapest) for the generation step.
- Keep context to 1–3 sentences. Long contexts dilute the chunk itself in the embedding.
- Be explicit in the prompt that context is for *search*, not for the model to use as the answer.
- Test on a representative sample of chunks; some chunks (e.g. tables of contents) get weird contexts.
- Re-generate when the document changes; cache aggressively in your generation pipeline.

## Three real-world scenarios

**Scenario 1: The legal research tool that finally worked.**
A team built a legal research tool with naive RAG over 50K court documents. Retrieval failed on cross-referencing queries ("the case mentioned in the dissent of Smith v. Jones"). They added Contextual Retrieval — context for each chunk includes case name, year, court, section. Failures dropped 60%. They saved a query that was previously useless.

**Scenario 2: The customer support cost calc.**
A team estimated Contextual Retrieval generation costs for their 500K-token KB. Without caching: $40. With prompt caching: $0.40. They generated context for the whole KB during a coffee break and got a meaningful quality improvement for under a dollar.

**Scenario 3: The over-engineering trap.**
A team built Contextual Retrieval + BM25 + reranking before measuring baseline RAG. They had no idea which layer was helping. They tore it down, started from baseline, added one layer at a time with evals between. Found that for their corpus, contextual embeddings alone got them 90% of the way there. Reranking added 1% and 200ms of latency. They shipped the simpler version.

## Common mistakes to avoid

- **Building all layers without measuring.** Start with baseline RAG; add one layer at a time.
- **No prompt caching on context generation.** 100× more expensive than it needs to be.
- **Long-winded context.** Context is for retrieval, not narration. Keep it short.
- **Re-generating context unnecessarily.** It's deterministic; cache outputs by chunk hash.
- **Skipping BM25.** Especially for queries with proper nouns, codes, or rare terms — keyword wins where embeddings miss.
