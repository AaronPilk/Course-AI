# How Stripe sees the world — the data model

## The puzzle

Stripe's API has a hundred-plus objects: Charges, PaymentIntents, SetupIntents, Customers, Products, Prices, Subscriptions, Invoices, Disputes, Refunds, Sources, Tokens, PaymentMethods, BalanceTransactions, Payouts, and many more. Without a mental model, you read the docs and feel lost.

Underneath, there's a clean structure. Once you see it, the docs become navigable.

## The simple version

Stripe's core model has six objects you'll touch constantly:

1. **Customer** — a person or business you charge.
2. **PaymentMethod** — a card / bank account / wallet, attached to a customer.
3. **PaymentIntent** — a single payment attempt.
4. **Product** — a thing you sell ("Pro Plan").
5. **Price** — a specific price for a product ("$20/mo" or "$200/year").
6. **Subscription** — a recurring charge for a Customer at a Price.

Plus: **Invoice** (auto-generated by subscriptions), **Charge** (legacy; PaymentIntent generates one), **Refund**, **Dispute**.

That's most of what you need.

## The technical version

### Customer

A Customer represents who you're charging. Created when a user signs up or first pays:

```js
const customer = await stripe.customers.create({
  email: "alex@example.com",
  name: "Alex Chen",
  metadata: { internal_user_id: "u_123" }
});
// customer.id → "cus_NfwG..."
```

A Customer holds:

- Email, name, address, phone.
- Default payment method.
- Subscriptions.
- Saved payment methods.
- Metadata (arbitrary key-value for your app's needs).

Reference by ID (`cus_...`) everywhere. Store in your database alongside your user record.

### PaymentMethod

A PaymentMethod is a card, bank account, wallet, or other way to pay:

```js
// In your frontend with Stripe.js:
const { paymentMethod } = await stripe.createPaymentMethod({
  type: "card",
  card: cardElement
});
// paymentMethod.id → "pm_..."

// Attach to customer (often automatic with PaymentIntents):
await stripe.paymentMethods.attach(paymentMethod.id, { customer: customer.id });
```

PaymentMethods are reusable. Once attached to a Customer, you can charge them again without re-collecting card details.

### PaymentIntent — the modern payment primitive

A PaymentIntent represents one payment attempt, end-to-end:

```js
const paymentIntent = await stripe.paymentIntents.create({
  amount: 2000, // in cents — $20.00
  currency: "usd",
  customer: customer.id,
  payment_method: paymentMethod.id,
  confirm: true
});
```

The PaymentIntent goes through states:

- **`requires_payment_method`** — needs a card.
- **`requires_confirmation`** — has a method; needs you to confirm.
- **`requires_action`** — 3DS / customer action required.
- **`processing`** — being processed.
- **`succeeded`** — payment complete.
- **`requires_capture`** — for separate auth/capture flows.
- **`canceled`** — abandoned.

You react to state changes — typically via webhooks (Module 4).

### Products and Prices

For things you sell with stable pricing (subscriptions, one-time products):

```js
const product = await stripe.products.create({
  name: "Pro Plan"
});

const price = await stripe.prices.create({
  product: product.id,
  unit_amount: 2000, // $20.00
  currency: "usd",
  recurring: { interval: "month" }
});
```

A Product is an abstract thing ("Pro Plan"). A Price is how it's priced ($20/mo, $200/year, $0.10/unit). One Product can have many Prices.

### Subscription

A Subscription charges a Customer at a Price repeatedly:

```js
const subscription = await stripe.subscriptions.create({
  customer: customer.id,
  items: [{ price: price.id }]
});
```

Stripe handles the recurring billing, generates invoices, attempts payment, and emits webhooks for each state change.

### Invoice

Subscriptions auto-create Invoices. An Invoice is "here's what we're billing for this period." You can also create standalone Invoices for one-off bills:

```js
const invoice = await stripe.invoices.create({
  customer: customer.id,
  collection_method: "send_invoice",
  days_until_due: 30
});

await stripe.invoiceItems.create({
  customer: customer.id,
  invoice: invoice.id,
  amount: 5000,
  currency: "usd",
  description: "Consulting hours"
});

await stripe.invoices.sendInvoice(invoice.id);
```

### Charge (legacy)

The old payment primitive. A PaymentIntent generates a Charge under the hood when it succeeds. You rarely create Charges directly anymore — Stripe recommends PaymentIntents for new integrations.

You'll see Charge in older code and the dashboard.

### Refund

Issued against a successful PaymentIntent or Charge:

```js
const refund = await stripe.refunds.create({
  payment_intent: paymentIntent.id,
  amount: 1000, // partial; omit for full
  reason: "requested_by_customer"
});
```

### Dispute

When a customer disputes a charge with their bank. Module 5.

### How they fit together

A typical subscription flow:

```
1. Customer signs up → create Customer
2. Customer enters card → create PaymentMethod, attach to Customer
3. Customer subscribes to Pro Plan → create Subscription using Price
4. Stripe generates Invoice for first period
5. Stripe creates PaymentIntent against Customer's default PaymentMethod
6. PaymentIntent succeeds → generates Charge
7. Webhook fires (invoice.paid, customer.subscription.created)
8. Your app marks user as Pro
9. Next month: Stripe repeats steps 4-7 automatically
```

Each object plays its role; the API stitches them together.

### Metadata — your secret weapon

Every Stripe object has a `metadata` field — arbitrary string key-value pairs:

```js
{
  metadata: {
    internal_user_id: "u_123",
    plan_tier: "pro",
    referred_by: "u_456"
  }
}
```

Use metadata to link Stripe objects to your database. Without it, reconciling Stripe events to your users is painful.

Don't put secrets in metadata — it's visible in the dashboard and logs.

### IDs and prefixes

Every Stripe ID has a prefix telling you its type:

- `cus_` — Customer.
- `pi_` — PaymentIntent.
- `pm_` — PaymentMethod.
- `prod_` — Product.
- `price_` — Price.
- `sub_` — Subscription.
- `inv_` — Invoice.
- `ch_` — Charge.
- `re_` — Refund.
- `evt_` — Event (webhook).
- `whsec_` — Webhook signing secret.

Once you internalize prefixes, scanning logs is much faster.

### Test mode vs live mode

Every object is either in test mode or live mode. Two separate worlds:

- **Test IDs** have different prefixes (often `_test_` somewhere, or just different IDs).
- **Test API keys** start with `sk_test_` / `pk_test_`.
- **Live API keys** start with `sk_live_` / `pk_live_`.

Never mix them. Test cards don't work in live mode; live cards don't work in test mode. Lesson 1.3 goes deep.

## Three real-world scenarios

**Scenario 1: The model click.**
A developer struggling with Stripe spent an hour drawing the data model on paper. Customer → PaymentMethod → PaymentIntent → Charge. Products → Prices → Subscriptions → Invoices → PaymentIntents. After that hour, the docs made sense. Worth the time investment.

**Scenario 2: The missing metadata.**
A team didn't put internal user IDs in Stripe metadata. When a payment failed, they couldn't easily figure out which user to notify — they had to look up by email. Once they added `metadata.user_id`, reconciliation became one-step.

**Scenario 3: The Charge confusion.**
A new team started with old Stripe tutorials using Charges directly. They had to refactor to PaymentIntents when 3D Secure became mandatory in some regions. Lesson: use PaymentIntents from day 1.

## Common mistakes to avoid

- **Skipping the data model.** Random Stripe code without understanding what an object is.
- **No metadata.** Reconciliation becomes painful.
- **Mixing test and live keys.** Recipe for incidents.
- **Direct Charges instead of PaymentIntents.** Legacy pattern; doesn't support 3DS cleanly.
- **One Price per Product mindset.** Products and Prices are separate for a reason.

## Summary

- Six core objects: **Customer, PaymentMethod, PaymentIntent, Product, Price, Subscription** + Invoice, Refund, Dispute.
- **PaymentIntents replaced Charges** as the modern payment primitive.
- **Products and Prices are separate** — one product can have many prices.
- **Metadata** links Stripe objects to your DB.
- **ID prefixes** tell you object type at a glance.
- **Test and live modes are entirely separate** — never mix keys.
