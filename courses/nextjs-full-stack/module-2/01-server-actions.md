# Server Actions explained

## What they are

Server Actions are async functions that run on the server, callable from Client Components as if they were local. You write the function once, mark it with `'use server'`, and call it from your form or button. Next.js handles the RPC mechanics: serialization, transport, error handling.

This replaces a lot of the REST/tRPC plumbing apps used to need for mutations. No `/api/posts/create` endpoint, no fetch wrapper, no type duplication. Just a server function called from client code.

```tsx
// app/posts/actions.ts
'use server';

import { db } from '@/lib/db';
import { revalidatePath } from 'next/cache';

export async function createPost(formData: FormData) {
  const title = formData.get('title') as string;
  const body = formData.get('body') as string;
  
  await db.posts.create({ data: { title, body } });
  revalidatePath('/posts');
}
```

```tsx
// app/posts/new/page.tsx
import { createPost } from '../actions';

export default function NewPostPage() {
  return (
    <form action={createPost}>
      <input name="title" />
      <textarea name="body" />
      <button>Create</button>
    </form>
  );
}
```

That's a complete mutation. No API route, no fetch, no boilerplate.

## How it works under the hood

When a Server Action is called from a Client Component:

1. Next.js serializes the arguments.
2. POSTs them to a hidden endpoint (auto-generated by the framework).
3. The server executes the action function.
4. Returns the result (or error).
5. Next.js deserializes and gives it back to the caller.

You don't see any of this. The function call feels local; the network transport is automatic. Types flow through — the client knows the action's signature without duplication.

## Calling from forms

The simplest pattern: form `action={action}`:

```tsx
<form action={createPost}>
  <input name="title" />
  <button>Save</button>
</form>
```

When the form submits:

1. Browser collects form data.
2. If JavaScript is available, it submits via Server Action.
3. If JavaScript is unavailable, the form submits as a regular HTML POST (progressive enhancement).

The progressive enhancement is huge: your forms work even when JavaScript fails. Users on slow connections, broken JS, or disabled JS still complete actions.

## Calling from event handlers

For non-form interactions (button clicks, programmatic calls):

```tsx
'use client';
import { deletePost } from '../actions';

export function DeleteButton({ id }: { id: string }) {
  return (
    <button onClick={() => deletePost(id)}>
      Delete
    </button>
  );
}
```

Same function; called directly. No FormData involved.

For passing arguments to form-action calls (since FormData is the default), bind:

```tsx
'use client';
import { deletePost } from '../actions';

export function DeleteForm({ id }: { id: string }) {
  const deleteWithId = deletePost.bind(null, id);
  return (
    <form action={deleteWithId}>
      <button>Delete</button>
    </form>
  );
}
```

`bind` pre-fills arguments. Server-side, the action receives `(id, formData)`.

## Validating input

Server Actions can validate inputs and return errors:

```tsx
'use server';
import { z } from 'zod';

const PostSchema = z.object({
  title: z.string().min(1).max(200),
  body: z.string().min(1),
});

export async function createPost(prevState: any, formData: FormData) {
  const result = PostSchema.safeParse({
    title: formData.get('title'),
    body: formData.get('body'),
  });
  
  if (!result.success) {
    return { error: result.error.flatten() };
  }
  
  await db.posts.create({ data: result.data });
  revalidatePath('/posts');
  return { success: true };
}
```

Pair with `useFormState` in the form to display errors. Covered in the next lesson.

## Revalidation

After a mutation, the data shown elsewhere on the page is now stale. Two revalidation tools:

**`revalidatePath(path)`** — invalidates the cache for a specific path:

```tsx
revalidatePath('/posts');         // Invalidates /posts page
revalidatePath('/posts/[slug]');  // Invalidates dynamic /posts/[slug] pages
revalidatePath('/', 'layout');    // Invalidates everything under the layout
```

**`revalidateTag(tag)`** — invalidates fetches tagged with a specific key:

```tsx
revalidateTag('posts');  // Invalidates any fetch that had next: { tags: ['posts'] }
```

After the action, the next render uses fresh data.

## Redirects

Server Actions can redirect after success:

```tsx
'use server';
import { redirect } from 'next/navigation';

export async function createPost(formData: FormData) {
  const post = await db.posts.create({ data: { ... } });
  redirect(`/posts/${post.id}`);
}
```

The client receives the redirect; the browser navigates. No need to manage redirect URLs from client state.

## Authentication in actions

Server Actions can use cookies, headers, and any server-side auth check:

```tsx
'use server';
import { cookies } from 'next/headers';

export async function createPost(formData: FormData) {
  const session = cookies().get('session');
  if (!session) throw new Error('Unauthorized');
  
  const user = await getUserFromSession(session.value);
  if (!user) throw new Error('Invalid session');
  
  await db.posts.create({ data: { ...formData, authorId: user.id } });
}
```

Always validate auth inside the action. Don't trust the client to gate access — the server is the security boundary.

## Sharing actions across files

Server Actions can live in their own files for reuse:

```tsx
// app/lib/actions.ts
'use server';

export async function createPost(formData: FormData) { /* ... */ }
export async function deletePost(id: string) { /* ... */ }
```

Import from anywhere:

```tsx
import { createPost } from '@/lib/actions';
```

Or define them inline in Server Components:

```tsx
export default async function Page() {
  async function action() {
    'use server';
    // ...
  }
  return <form action={action}>...</form>;
}
```

Co-located actions work well for one-off mutations specific to a page.

## Type safety

Server Actions are fully typed. Client code that calls an action sees:

- The exact argument types (TypeScript propagates).
- The return type.
- Any thrown errors.

No code generation, no schema validation step (beyond runtime validation if you want it). The function signature is the contract.

## Limitations

- **No streaming responses.** Actions return one result; no streaming for chat-style UIs (use API routes or external services for that).
- **Sequential.** Multiple parallel actions queue (or you have to manage parallelism yourself).
- **Cookie/header mutations.** Server Actions can set cookies and headers but not in all contexts; check docs for what works.
- **Files are mostly via FormData.** For complex multi-step uploads, REST endpoints may fit better.

For 90% of mutations, Server Actions are the cleanest path. The exceptions are niche.

## Comparison to other patterns

**vs REST API routes:**
- Server Actions: less code, types flow, no API surface to maintain.
- REST: needed for external consumers, webhooks, programmatic access.

**vs tRPC:**
- Server Actions: built-in, no extra library.
- tRPC: more features around queries (subscriptions, etc.), works across server/client boundaries more flexibly.

**vs traditional `fetch()` in event handlers:**
- Server Actions: type-safe, progressive enhancement, automatic revalidation.
- fetch: more flexible for non-mutation calls.

For typical app mutations in Next.js: use Server Actions.

## Mistakes to avoid

- **No auth check in the action.** Anyone can call it.
- **Forgetting `'use server'`.** Functions ship to the client; security issue.
- **No `revalidatePath`/`revalidateTag` after mutation.** Stale data displayed.
- **Returning sensitive data unnecessarily.** It crosses to the client; only return what's needed.
- **Catching all errors silently.** Failed mutations look successful.

## Summary

- Server Actions = `'use server'` async functions; called from client as if local.
- Forms via `action={fn}`; progressive enhancement built in.
- Pair with `useFormState` for validation errors.
- Use `revalidatePath` / `revalidateTag` after mutations.
- Auth-check inside the action; don't trust the client.
- Replace most REST mutation endpoints; keep REST for external consumers.
