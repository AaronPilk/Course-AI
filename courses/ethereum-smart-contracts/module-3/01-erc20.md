# ERC-20 fungible tokens

## What ERC-20 defines

Standard interface for fungible tokens:

```solidity
interface IERC20 {
    function totalSupply() external view returns (uint256);
    function balanceOf(address account) external view returns (uint256);
    function transfer(address to, uint256 amount) external returns (bool);
    function allowance(address owner, address spender) external view returns (uint256);
    function approve(address spender, uint256 amount) external returns (bool);
    function transferFrom(address from, address to, uint256 amount) external returns (bool);

    event Transfer(address indexed from, address indexed to, uint256 value);
    event Approval(address indexed owner, address indexed spender, uint256 value);
}
```

Adopted by every fungible token on Ethereum (USDC, DAI, USDT, WETH, etc.).

For: token interoperability.

## OpenZeppelin ERC20

```solidity
import "@openzeppelin/contracts/token/ERC20/ERC20.sol";

contract MyToken is ERC20 {
    constructor(uint256 initialSupply) ERC20("MyToken", "MTK") {
        _mint(msg.sender, initialSupply);
    }
}
```

That's it. Deploy gives you a fully-functional ERC-20.

For: production launches.

## Decimals

```solidity
function decimals() public pure override returns (uint8) {
    return 6;    // USDC-style 6 decimals
}
```

Default 18 (matching ETH). USDC / USDT use 6. Display amount = on-chain / 10^decimals.

For: human-readable amounts.

## Mint + burn

```solidity
import "@openzeppelin/contracts/token/ERC20/extensions/ERC20Burnable.sol";
import "@openzeppelin/contracts/access/Ownable.sol";

contract Token is ERC20, ERC20Burnable, Ownable {
    constructor() ERC20("X", "X") Ownable(msg.sender) {}

    function mint(address to, uint256 amount) external onlyOwner {
        _mint(to, amount);
    }
}
```

For: controlled supply.

## Approve + transferFrom flow

```
1. User calls token.approve(spender, 100)
2. Spender calls token.transferFrom(user, recipient, 100)
3. Token deducts user's balance + allowance
```

DeFi: user approves DEX → DEX pulls tokens during swap.

For: delegated spending.

## Approve race condition

Old hazard: setting allowance from N → M lets attacker drain N then receive M.

Mitigation:
```solidity
// Use increaseAllowance / decreaseAllowance
token.increaseAllowance(spender, 50);
token.decreaseAllowance(spender, 30);

// Or set to 0 first
token.approve(spender, 0);
token.approve(spender, newAmount);
```

For: safe allowance management.

## permit (ERC-2612)

```solidity
import "@openzeppelin/contracts/token/ERC20/extensions/ERC20Permit.sol";

contract Token is ERC20, ERC20Permit {
    constructor() ERC20("X", "X") ERC20Permit("X") {}
}
```

Off-chain signed approval; no separate approve tx:
```
1. User signs permit message off-chain
2. DEX submits permit + transferFrom in single tx
3. No prior on-chain approval needed
```

For: 1-tx UX.

## Pausable

```solidity
import "@openzeppelin/contracts/token/ERC20/extensions/ERC20Pausable.sol";

contract Token is ERC20, ERC20Pausable, Ownable {
    function pause() external onlyOwner { _pause(); }
    function unpause() external onlyOwner { _unpause(); }
}
```

Halt transfers during incident.

For: emergency controls.

## Snapshots

```solidity
import "@openzeppelin/contracts/token/ERC20/extensions/ERC20Snapshot.sol";

contract Token is ERC20, ERC20Snapshot {
    function snapshot() external onlyOwner returns (uint256) {
        return _snapshot();
    }

// Read balance at specific snapshot
uint256 balance = token.balanceOfAt(user, snapshotId);
```

For: airdrops based on historic holdings.

## Common mistakes

- **Returning false instead of revert.** SafeERC20 wraps in revert-on-false.
- **Approve race exploit.** Use increase/decreaseAllowance.
- **Fee-on-transfer tokens.** Some tokens (deflationary) take fee on transfer; check actual received.
- **Non-standard tokens.** USDT didn't return bool until 2022.

```solidity
import "@openzeppelin/contracts/token/ERC20/utils/SafeERC20.sol";

using SafeERC20 for IERC20;
IERC20(token).safeTransfer(to, amount);    // Reverts on failure
IERC20(token).safeApprove(spender, amount); // Handles weird tokens
```

For: safe token integration.

## Gas-optimized ERC-20

For high-volume tokens (DEX rewards, etc.):
- Solady's ERC20 implementation (cheaper than OpenZeppelin).
- Custom variants for fixed supply / non-burnable.
- Skipping unused features.

For: gas-critical use cases.

## Token deployer pattern

```solidity
contract TokenFactory {
    event TokenCreated(address indexed token, address indexed creator);

    function createToken(string memory name, string memory symbol, uint256 supply)
        external returns (address)
    {
        Token token = new Token(name, symbol, supply, msg.sender);
        emit TokenCreated(address(token), msg.sender);
        return address(token);
    }
}
```

Many tokens from one factory.

For: launchpad architecture.

## Mistakes to avoid

- **Custom ERC-20 from scratch.** Use OpenZeppelin; audited.
- **Not handling non-standard tokens.** USDT / weird tokens; use SafeERC20.
- **Renouncing ownership during testing.** Locked out.
- **No max supply cap on minting.** Investor distrust.

## Summary

- ERC-20 = standard fungible token interface.
- OpenZeppelin = audited reference implementation.
- approve + transferFrom for delegated spending.
- permit (ERC-2612) for 1-tx UX.
- SafeERC20 handles non-standard tokens.
