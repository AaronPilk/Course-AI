# R2 — S3-compatible object storage

## What R2 is

Cloudflare R2 is object storage — like AWS S3 but with no egress fees and integrated with Workers.

The defining feature: **zero egress fees**. Reading data out of R2 to your users (or to other clouds) costs nothing. AWS S3 charges $0.05-0.09 per GB egress; at scale this adds up to thousands of dollars per month.

R2 also has cheaper storage ($0.015/GB/month vs S3's $0.023) and competitive operation pricing.

For any workload moving meaningful data, R2's economics dominate.

## API

R2 is accessible two ways:

**S3-compatible API:** existing S3 tools and SDKs work. Point them at R2's endpoint, set the right credentials.

**Workers binding:** type-safe access from Workers.

```ts
type Env = { ASSETS: R2Bucket };

async fetch(request: Request, env: Env) {
  // GET an object:
  const obj = await env.ASSETS.get('photo.jpg');
  if (!obj) return new Response('Not found', { status: 404 });
  return new Response(obj.body, {
    headers: { 'Content-Type': obj.httpMetadata?.contentType || 'application/octet-stream' },
  });
}
```

`obj.body` is a ReadableStream — pipe directly to Response. No buffering in Worker memory; large files stream through.

## Common operations

```ts
// PUT an object:
await env.ASSETS.put('photo.jpg', request.body, {
  httpMetadata: { contentType: 'image/jpeg' },
  customMetadata: { uploadedBy: userId },
});

// DELETE:
await env.ASSETS.delete('photo.jpg');

// List:
const result = await env.ASSETS.list({ prefix: 'users/123/', limit: 100 });
for (const obj of result.objects) {
  console.log(obj.key, obj.size, obj.uploaded);
}

// Head (metadata without body):
const meta = await env.ASSETS.head('photo.jpg');
console.log(meta?.size, meta?.httpMetadata);
```

Same operations as S3; same mental model.

## Use cases

**User uploads.** Profile pictures, document uploads, file sharing.

**Static assets.** Images, video, downloads. Pair with Cloudflare CDN for caching.

**Backups.** Database dumps, log archives — zero-egress means cheap restores.

**Build artifacts.** Compiled outputs, container images, release files.

**Data lake style.** Logs and analytics dumped for processing.

For anything bigger than ~25MB (KV's limit), R2.

## Public access

Make objects publicly accessible:

```ts
// In wrangler.toml or via dashboard:
// Custom domain attached to R2 bucket; objects served at:
// https://files.example.com/photo.jpg
```

Or generate signed URLs for time-limited access:

```ts
import { sign } from '@cloudflare/r2-presigned-url';

const url = await sign({
  bucket: 'my-assets',
  key: 'photo.jpg',
  expiresIn: 3600,  // 1 hour
});
```

(R2 supports presigned URLs via S3-compatible API; SDKs help generate.)

## Streaming uploads

For large uploads, stream to R2 directly:

```ts
async fetch(request: Request, env: Env) {
  if (request.method !== 'PUT') return new Response('Method not allowed', { status: 405 });
  
  await env.ASSETS.put('upload.bin', request.body, {
    httpMetadata: {
      contentType: request.headers.get('content-type') || 'application/octet-stream',
    },
  });
  
  return new Response('Uploaded');
}
```

`request.body` is a ReadableStream; piped directly to R2. No memory buffering; supports arbitrarily large uploads.

For multi-GB uploads, S3 multipart upload protocol is supported.

## Conditional operations

```ts
// Only put if doesn't exist:
const result = await env.ASSETS.put('file.txt', body, {
  onlyIf: { etagDoesNotMatch: '*' },  // any value
});

// Only get if changed:
const obj = await env.ASSETS.get('file.txt', {
  onlyIf: { etagDoesNotMatch: previousEtag },
});
```

ETags enable cache-friendly serving and concurrency control.

## R2 + Workers caching

For hot files, combine R2 with Workers' Cache API:

```ts
async fetch(request: Request, env: Env, ctx: ExecutionContext) {
  const url = new URL(request.url);
  const cacheKey = new Request(url.toString(), request);
  const cache = caches.default;
  
  let response = await cache.match(cacheKey);
  if (!response) {
    const obj = await env.ASSETS.get(url.pathname.slice(1));
    if (!obj) return new Response('Not found', { status: 404 });
    
    response = new Response(obj.body, {
      headers: {
        'Content-Type': obj.httpMetadata?.contentType || 'application/octet-stream',
        'Cache-Control': 'public, max-age=86400',
      },
    });
    ctx.waitUntil(cache.put(cacheKey, response.clone()));
  }
  
  return response;
}
```

First request fetches from R2 + caches at edge; subsequent requests from edge cache (microseconds). Cache invalidates per `Cache-Control` headers.

## Image transforms

R2 + Cloudflare Image Resizing:

```ts
// URL: /image.jpg?width=400&format=webp
// Cloudflare automatically resizes/converts on the fly
```

Built into Cloudflare; works with R2-backed images. Resize, format conversion, quality adjustment via URL params. Eliminates server-side image pipelines.

For more control: Workers can apply transforms via Image API or WASM libraries.

## Lifecycle policies

R2 supports object expiration via lifecycle rules:

```bash
# Via wrangler or dashboard:
wrangler r2 bucket lifecycle add my-bucket \
  --rule '{"id": "delete-old", "filter": {"prefix": "logs/"}, "expirationDays": 30}'
```

Logs deleted after 30 days; backups expire automatically; temp files clean themselves. Saves storage costs.

## S3 compatibility

Use existing S3 SDKs:

```ts
import { S3Client, PutObjectCommand } from '@aws-sdk/client-s3';

const client = new S3Client({
  region: 'auto',
  endpoint: `https://${env.ACCOUNT_ID}.r2.cloudflarestorage.com`,
  credentials: {
    accessKeyId: env.R2_ACCESS_KEY,
    secretAccessKey: env.R2_SECRET_KEY,
  },
});

await client.send(new PutObjectCommand({
  Bucket: 'my-bucket',
  Key: 'file.txt',
  Body: 'content',
}));
```

Useful when:
- Migrating from S3; existing code works.
- Using libraries that expect S3 (sharp, AWS-SDK-based tools).
- Accessing R2 from outside Workers.

For new Workers code, prefer R2 bindings (simpler, faster).

## R2 + D1 + Workers pattern

A common architecture:

```
Worker
├── D1 (metadata: which files exist, who owns them, when uploaded)
└── R2 (the actual file contents)
```

D1 query: "find all files belonging to user X."
R2 fetch: "serve the file at path Y."

Together: relational queries + cheap large-object storage.

## Limits

- **Object size:** 5TB max per object.
- **Bucket count:** 1000 per account (typically).
- **Operations:** $4.50 per million Class A (writes); $0.36 per million Class B (reads). Free tier generous.
- **Bandwidth out:** $0. The defining advantage.

For most apps: storage cost dominates; the math beats S3 substantially.

## Migration from S3

R2 supports "sippy" — read-through from S3:

1. Point R2 at your existing S3 bucket.
2. Reads check R2 first; on miss, fetch from S3 and store in R2.
3. Over time, hot data ends up in R2.
4. Stop writing to S3; eventually delete.

Zero-downtime migration; reads cost less over time as cache fills.

## Mistakes to avoid

- **Treating R2 as a database.** No querying; pair with D1 for metadata.
- **No cache headers.** Each request goes back to R2 storage; slower + costlier.
- **No lifecycle policies on logs.** Storage costs grow forever.
- **Storing thousands of small objects when one would do.** Operation cost adds up.
- **Public buckets for sensitive data.** Use signed URLs or auth-gated Workers.

## Summary

- R2 is S3-compatible object storage with zero egress fees.
- Bindings give Workers type-safe access; S3 SDKs also work.
- Stream uploads and downloads via ReadableStream.
- Pair with Cache API for edge-cached files.
- Image Resizing + R2 = built-in image transformation pipeline.
- Lifecycle policies auto-expire old objects.
- D1 for metadata + R2 for file contents = clean architecture.
