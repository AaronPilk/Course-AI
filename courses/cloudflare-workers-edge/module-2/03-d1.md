# D1 — SQLite at the edge

## What D1 is

D1 is SQLite as a service from Cloudflare. Full SQL — JOINs, transactions, indexes — running globally with read replicas at the edge.

For Workers needing relational data, D1 is the native answer. No connection pooling concerns; no region selection; bindings just work.

## Setup

```bash
# Create a database:
wrangler d1 create my-database
```

This outputs binding config:

```toml
# wrangler.toml
[[d1_databases]]
binding = "DB"
database_name = "my-database"
database_id = "abc-def-..."
```

Create schema via migration files:

```bash
wrangler d1 migrations create my-database initial_schema
```

Edit the generated migration file:

```sql
-- migrations/0001_initial_schema.sql
CREATE TABLE users (
  id TEXT PRIMARY KEY,
  email TEXT NOT NULL UNIQUE,
  created_at INTEGER NOT NULL DEFAULT (unixepoch())
);

CREATE INDEX users_email_idx ON users(email);
```

Apply migrations:

```bash
wrangler d1 migrations apply my-database --local      # to local dev DB
wrangler d1 migrations apply my-database --remote     # to remote DB
```

## Querying

```ts
type Env = { DB: D1Database };

async fetch(request: Request, env: Env) {
  // Prepared statement:
  const { results } = await env.DB.prepare('SELECT * FROM users WHERE id = ?')
    .bind('user-123')
    .all();
  
  return Response.json(results);
}
```

Always use prepared statements with parameters. Never concatenate user input into SQL — SQL injection is a real risk.

Multiple ways to read:

```ts
// All rows:
const { results } = await stmt.all<User>();

// First row:
const user = await stmt.first<User>();

// First column of first row (e.g., COUNT(*)):
const count = await stmt.first<number>('count');

// Run without expecting results (UPDATE/DELETE):
const { meta } = await stmt.run();
console.log(meta.changes);  // rows affected
```

## Batch queries

For multiple queries:

```ts
const result = await env.DB.batch([
  env.DB.prepare('UPDATE users SET tier = ? WHERE id = ?').bind('pro', userId),
  env.DB.prepare('INSERT INTO upgrades (user_id, date) VALUES (?, ?)').bind(userId, Date.now()),
]);
```

Batch runs queries together — same transaction semantics where applicable; fewer round-trips.

## Transactions

For multi-step atomic operations, use batch or transactions:

```ts
await env.DB.batch([
  env.DB.prepare('BEGIN'),
  env.DB.prepare('UPDATE accounts SET balance = balance - ? WHERE id = ?').bind(100, senderId),
  env.DB.prepare('UPDATE accounts SET balance = balance + ? WHERE id = ?').bind(100, recipientId),
  env.DB.prepare('COMMIT'),
]);
```

For more complex transactions, use the typescript ORM Drizzle (covered next section) which has transaction helpers.

## Migrations

D1 migration files run in order:

```
migrations/
├── 0001_initial_schema.sql
├── 0002_add_posts_table.sql
└── 0003_add_user_role.sql
```

Wrangler tracks which have been applied. Apply forward; D1 doesn't have automatic down-migrations (write a new migration to undo).

In production:

```bash
wrangler d1 migrations apply my-database --remote
```

For Workers projects with continuous deployment, run migrations via CI as part of the deploy pipeline.

## D1 with Drizzle ORM

Raw SQL works; Drizzle ORM makes it type-safe:

```ts
import { drizzle } from 'drizzle-orm/d1';
import { sql, eq } from 'drizzle-orm';
import { sqliteTable, text, integer } from 'drizzle-orm/sqlite-core';

const users = sqliteTable('users', {
  id: text('id').primaryKey(),
  email: text('email').notNull().unique(),
  createdAt: integer('created_at').default(sql`(unixepoch())`),
});

async fetch(request: Request, env: Env) {
  const db = drizzle(env.DB);
  
  const allUsers = await db.select().from(users).all();
  const oneUser = await db.select().from(users).where(eq(users.id, '123')).get();
  
  await db.insert(users).values({ id: '123', email: 'a@b.com' }).run();
}
```

Schema-as-code; type-safe queries; standard patterns. Drizzle is the modern ORM choice for Workers + D1.

## Performance considerations

D1 is fast but has nuances:

- **Reads:** can be served from regional read replicas; sub-ms latency at edge after warm-up.
- **Writes:** go to a primary region; cross-region latency for writes.
- **Indexes are essential.** Same as any SQLite; missing indexes = full table scans.
- **Don't use D1 for OLAP.** It's transactional/OLTP-shaped.

For analytics or large aggregations, point to a separate analytics store. D1 is for the typical operational data.

## Limits

- **Database size:** 10GB per database (beta limits; growing).
- **Query timeout:** typically a few seconds.
- **Rows per query:** large but practical limits.
- **Free tier:** 5 million rows reads/day; 100k writes/day. Paid tiers higher.

Stay within reasonable scale (millions of rows, not billions). For larger, consider sharding or moving to Postgres + Hyperdrive.

## Local development

`wrangler dev` spins up a local SQLite for D1:

```bash
wrangler dev               # uses local D1 by default
wrangler dev --remote      # uses production D1
```

Local D1 is a real SQLite file; queries behave like production. Useful for fast iteration.

For testing migrations:

```bash
wrangler d1 migrations apply my-database --local
```

## When to use D1 vs alternatives

**D1:**
- Relational data with joins, indexes.
- Up to ~10GB total data.
- Read-heavy workloads.
- OLTP shape.

**KV:**
- Simple key-value.
- High read throughput.
- Eventually consistent.

**Hyperdrive + Postgres:**
- Larger than D1's limits.
- PostGIS, JSONB, full-text search.
- Existing Postgres data.

**Durable Objects:**
- Coordination (chat, games).
- Strong consistency per object.
- Lower-level than D1.

**R2:**
- Files / large objects.
- 5TB per object.

Most apps use D1 for primary data + KV for caches + R2 for files.

## Common queries

```sql
-- Search by indexed field:
SELECT * FROM users WHERE email = ? LIMIT 1;

-- Join:
SELECT p.*, u.name as author_name
FROM posts p
JOIN users u ON u.id = p.author_id
WHERE p.published = 1
ORDER BY p.created_at DESC
LIMIT 20;

-- Aggregation:
SELECT category, COUNT(*) as count
FROM products
WHERE in_stock = 1
GROUP BY category;

-- Window function (modern SQLite):
SELECT name, score, RANK() OVER (ORDER BY score DESC) as rank
FROM leaderboard;
```

SQLite supports all standard SQL; D1 exposes the full surface. Familiar patterns translate directly.

## Mistakes to avoid

- **String concatenation in queries.** SQL injection.
- **No indexes on filter/sort columns.** Slow at scale.
- **Treating D1 as a cache.** Use KV.
- **Storing huge BLOBs in D1.** Use R2 for files; reference by ID.
- **No migration discipline.** Schema drift between environments.

## Summary

- D1 = managed SQLite at the edge with global read replicas.
- Bindings provide type-safe Worker access.
- Migrations via SQL files; wrangler applies them.
- Drizzle ORM for type-safe schema and queries.
- Up to ~10GB per database; OLTP-shaped.
- For larger data: Hyperdrive + external Postgres.
- Common architecture: D1 for primary + KV for caches + R2 for files.
