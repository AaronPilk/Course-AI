# KV — fast eventually-consistent key-value

## What KV is

Cloudflare Workers KV is a global key-value store optimized for high read throughput. Writes propagate to all edge locations within ~60 seconds; reads happen from the nearest cache (microseconds at edge after first hit).

Think of it as a globally-distributed cache that supports writes. Not a database — there's no querying beyond `get(key)`, no transactions, no strong consistency.

## The right shape for KV

KV fits:

- **Session tokens** — keyed by token; lookups every request.
- **Feature flags** — keyed by user/account; rarely written, frequently read.
- **Rate limits** — counters per IP/key (with caveats; see below).
- **Configuration** — keyed by tenant/env; updates rare, reads constant.
- **A/B test buckets** — keyed by user; persistent assignment.
- **Computed caches** — expensive results keyed by input.

KV doesn't fit:

- **Transactional data** — no transactions; eventually consistent.
- **Frequent counter updates** — write-throughput limits.
- **Queryable data** — only `get(key)`; no scans by attribute.
- **Strong consistency requirements** — KV is eventually consistent.

## API

```ts
// Read:
const value = await env.MY_KV.get('user:123');
// → string | null

// Read JSON:
const obj = await env.MY_KV.get('user:123', 'json');
// → any | null (parsed)

// Read with metadata:
const { value, metadata } = await env.MY_KV.getWithMetadata('user:123');

// Write:
await env.MY_KV.put('user:123', JSON.stringify({ name: 'Aaron' }));

// Write with TTL:
await env.MY_KV.put('session:abc', token, { expirationTtl: 3600 }); // 1 hour

// Write with metadata:
await env.MY_KV.put('user:123', value, {
  metadata: { tier: 'pro' },
});

// Delete:
await env.MY_KV.delete('user:123');

// List keys (slow; rate-limited):
const result = await env.MY_KV.list({ prefix: 'user:' });
```

## Consistency model

KV is **eventually consistent**:

- A write may take up to 60 seconds to be visible in all edge locations.
- Reads return whatever's in the nearest cache; may be stale.
- For "I just wrote this; can I read it back?" pattern: works after a few seconds; not guaranteed instant.

If you need read-after-write consistency, use Durable Objects or D1 (covered in next lessons).

## TTLs

Use TTLs for expiring data:

```ts
await env.MY_KV.put('session:abc', token, { expirationTtl: 3600 }); // 1 hour
await env.MY_KV.put('cache:foo', result, { expiration: Math.floor(Date.now()/1000) + 86400 }); // 24h from now
```

`expirationTtl` is relative seconds; `expiration` is absolute Unix timestamp. Both result in the key auto-deleting at that time.

Useful for caches and time-bounded data (sessions, rate limit windows).

## Limits

- **Value size:** 25MB max.
- **Key size:** 512 bytes.
- **List operations:** rate-limited; not for hot paths.
- **Write rate:** 1 write per key per second sustained; bursts higher.
- **Reads:** essentially unlimited.

For high-write workloads, KV is not the right tool. Durable Objects or D1 fit better.

## Patterns

**Session storage:**

```ts
async function getUser(token: string, env: Env): Promise<User | null> {
  const data = await env.SESSIONS.get(`token:${token}`, 'json');
  return data as User | null;
}

async function createSession(token: string, user: User, env: Env): Promise<void> {
  await env.SESSIONS.put(`token:${token}`, JSON.stringify(user), {
    expirationTtl: 24 * 60 * 60,
  });
}
```

Every request: cheap KV lookup; sub-millisecond. Sessions expire automatically.

**Feature flags:**

```ts
async function getFlags(userId: string, env: Env): Promise<Flags> {
  const cached = await env.FLAGS.get(`user:${userId}`, 'json');
  if (cached) return cached as Flags;
  
  const flags = await computeFlags(userId, env);
  await env.FLAGS.put(`user:${userId}`, JSON.stringify(flags), {
    expirationTtl: 60 * 60,
  });
  return flags;
}
```

Compute once, cache, serve from edge.

**Rate limiting (light):**

```ts
async function rateLimit(ip: string, env: Env): Promise<boolean> {
  const key = `rate:${ip}:${Math.floor(Date.now() / 60000)}`; // per-minute window
  const count = parseInt(await env.RATE_LIMITS.get(key) || '0');
  if (count >= 100) return false;
  await env.RATE_LIMITS.put(key, String(count + 1), { expirationTtl: 120 });
  return true;
}
```

Caveat: KV eventual consistency means rate limiting via KV is approximate. For strict limits, use Durable Objects.

## Caching expensive computation

```ts
async function getPriceForProduct(id: string, env: Env): Promise<Price> {
  const cacheKey = `price:${id}`;
  const cached = await env.CACHE.get(cacheKey, 'json');
  if (cached) return cached as Price;
  
  const price = await fetchPriceFromOrigin(id);
  await env.CACHE.put(cacheKey, JSON.stringify(price), {
    expirationTtl: 60,  // 1 minute
  });
  return price;
}
```

Combined with `ctx.waitUntil`, you can also refresh in background:

```ts
async function getCachedOrFresh(key: string, env: Env, ctx: ExecutionContext): Promise<T> {
  const cached = await env.CACHE.get(key, 'json');
  if (cached) {
    ctx.waitUntil(refresh(key, env));  // refresh in background
    return cached as T;
  }
  return fetchFresh(key, env);
}
```

Stale-while-revalidate pattern at edge.

## Listing keys

```ts
const result = await env.MY_KV.list({ prefix: 'user:', limit: 100 });
for (const key of result.keys) {
  console.log(key.name, key.metadata);
}
// Continue with cursor if more:
if (!result.list_complete) {
  const more = await env.MY_KV.list({ prefix: 'user:', cursor: result.cursor });
}
```

Lists are slower than gets. Use only for admin / cleanup operations. Don't list keys in user-request hot paths.

## When to use KV vs alternatives

**KV:**
- High read-to-write ratio.
- Eventual consistency OK.
- Simple key-value lookups.

**Durable Objects:**
- Strong consistency needed.
- Coordination required (chat rooms, game state).
- Frequent writes to same key.

**D1:**
- Relational data.
- Queries beyond by-key lookup.
- Transactions.

**R2:**
- Large objects (files, images).
- Larger than KV's 25MB limit.

**Cache API:**
- HTTP response caching tied to URLs.
- Lighter than KV for response caching.

Often you'll use multiple together: D1 as source of truth; KV as edge cache.

## Mistakes to avoid

- **Treating KV as a database.** No querying, no transactions, no strong consistency.
- **High write rates on same key.** Hits per-key rate limits.
- **Listing in hot paths.** Slow.
- **No TTLs on transient data.** Storage accumulates.
- **Storing huge values.** 25MB cap; R2 for larger.

## Summary

- KV: global key-value store optimized for reads.
- Eventually consistent; writes propagate within ~60s.
- Use for sessions, flags, caches, configs.
- Don't use for transactional or queryable data.
- TTLs auto-expire keys.
- Limits: 25MB values, ~1 write/sec per key.
- For strong consistency, use Durable Objects.
