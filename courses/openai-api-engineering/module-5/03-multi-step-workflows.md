# Multi-step workflows: deep research, computer use, voice agents

## The puzzle

So far, every pattern you've learned is *one model with one prompt*. Even agent loops are mostly one model calling tools.

The most ambitious AI products string many of these primitives together. A "deep research" agent isn't one model — it's a coordinator that fans out searches, summarizes findings, and writes a synthesis. A "computer use" agent isn't just a chat model — it's a perception loop that sees a screen, decides what to click, and verifies the result. A voice agent isn't an API call — it's a streaming session with interleaved tool use.

This lesson covers three of the most powerful multi-step patterns OpenAI's stack supports today: **deep research**, **computer use**, and **voice agents**. Each is a composition of primitives you already know, plus a few platform-specific pieces.

## The simple version

Three workflow patterns worth knowing:

1. **Deep research** — a model (or pipeline) that performs many searches, synthesizes a long answer with citations, and runs for minutes per query. Use for analytical work where depth matters more than speed.

2. **Computer use** — a model that sees screenshots of a desktop or browser and clicks/types like a human. Use for automating tasks across software that doesn't have a clean API.

3. **Voice agents** — bidirectional streaming voice powered by the Realtime API, often with tools. Use for phone-style interactive AI.

None of these are magic — they're orchestrations of primitives from earlier modules. But they have specific patterns and pitfalls worth knowing before you try to ship one.

## The technical version

### Deep research

The pattern: a high-quality long-form answer to a research question, generated by:

1. Decomposing the question into sub-queries.
2. Running web searches and reading results.
3. Following links, citing sources.
4. Synthesizing across many findings into a coherent report.
5. Returning the report (often markdown) with citations.

OpenAI offers a hosted deep-research capability that bundles this pattern. You can also build your own using the Responses API with web-search and URL-fetching tools.

Pattern, simplified:

```js
// Hosted deep research (when available in your model lineup)
const response = await openai.responses.create({
  model: "gpt-5-deep-research",  // or current deep-research model
  input: "What are the latest developments in fusion energy, and which companies are closest to commercialization?",
  tools: [{ type: "web_search" }, { type: "url_fetch" }]
});

console.log(response.output_text);
// Response includes citations as part of the structured output
```

Or roll your own with a reasoning model + tools:

```js
const tools = [
  { type: "function", function: { name: "web_search", parameters: {...} } },
  { type: "function", function: { name: "read_page", parameters: {...} } }
];

const response = await openai.responses.create({
  model: "gpt-5",
  reasoning: { effort: "high" },
  tools,
  input: "Research question..."
});
// Agent loops through tool calls until it has enough to answer.
```

Things to know:

- **Latency is minutes, not seconds.** Build UX around the wait — progress messages, stream intermediate steps if you can.
- **Cost per query is dollars, not cents.** Deep research uses reasoning tokens and many tool calls. Plan accordingly.
- **Citations matter.** Always surface sources to users. Without them, the output looks magical but isn't verifiable.
- **Caching helps.** If multiple users ask similar questions, cache reports keyed by canonicalized query.

Use cases:

- Competitive intelligence for sales teams.
- Investment research summaries.
- Literature reviews.
- "Tell me everything about X and cite your sources" features.

Misuses:

- Real-time UX. Reports take minutes.
- Simple factual lookups. Just call web search directly.
- High-volume cheap traffic. Each query is expensive.

### Computer use

The pattern: a model that takes a screenshot, decides on an action (click, type, scroll, wait), executes the action, and loops.

OpenAI's computer use models accept image inputs (screenshots) and output structured actions like `{type: "click", x: 320, y: 480}` or `{type: "type", text: "hello"}`. Your client code is responsible for actually executing those actions against the browser or OS.

Loop, at a high level:

```js
let context = { screenshot: await takeScreenshot(), goal: "Reserve a meeting room for tomorrow at 2pm" };

while (true) {
  const response = await openai.responses.create({
    model: "gpt-5-computer-use",
    input: [
      { role: "user", content: [
        { type: "input_text", text: `Goal: ${context.goal}` },
        { type: "input_image", image_url: `data:image/png;base64,${context.screenshot}` }
      ]}
    ],
    tools: [{ type: "computer_use" }]
  });

  const action = response.output[0];
  if (action.type === "done") break;

  await executeAction(action);  // your code: click, type, scroll, etc.
  context.screenshot = await takeScreenshot();
}
```

Things to know:

- **Sandboxed environments only.** Don't give a computer-use agent uncontrolled access to a real desktop. Use a VM, headless browser, or contained environment.
- **Confirmation steps for destructive actions.** Buying, deleting, submitting forms — gate behind approval (Module 3.5).
- **Error recovery is hard.** Models get confused when the UI changes mid-task. Build retries and timeouts.
- **Latency is high.** Each step is a model call plus a real UI action. Tasks that take a human 30 seconds might take an agent 2–3 minutes.
- **Test on real workflows.** Computer use looks great in demos and fails on subtle UI variations. Eval against your specific target apps.

Use cases:

- Automating workflows in legacy apps with no API.
- QA testing — agents that exercise UI flows.
- Browser-based data extraction at low volume.

Misuses:

- High-volume tasks where a proper API call would work.
- Mission-critical workflows where reliability matters more than flexibility.
- Anything sensitive without strict approval gates.

### Voice agents

The pattern: real-time streaming voice, both ways, often with tool use embedded. Built on the **Realtime API**.

Key properties:

- **Bidirectional streaming** — audio in, audio out, in real time over WebSocket or WebRTC.
- **Sub-second latency** — including model processing.
- **Voice options** — multiple speakers and voices, tunable.
- **Tool use mid-conversation** — the model can call functions while the user is talking.
- **Interrupt support** — the user can talk over the model, and the model handles it gracefully.

Architecture overview:

```js
// Establish a Realtime session
const session = await openai.realtime.create({
  model: "gpt-realtime",
  modalities: ["audio", "text"],
  voice: "alloy",
  tools: [
    { type: "function", function: { name: "lookup_appointment", ... } }
  ],
  instructions: "You are a friendly scheduling assistant..."
});

// Pipe audio in from the user's mic
session.send({ type: "input_audio_buffer.append", audio: micChunk });

// Receive audio back to play to the user
session.on("response.audio.delta", (chunk) => playAudio(chunk));

// Tool calls happen automatically in the session loop
session.on("response.function_call_arguments.done", async (call) => {
  const result = await runTool(call.name, call.arguments);
  session.send({ type: "response.function_call_output", call_id: call.id, output: result });
});
```

Things to know:

- **WebRTC vs. WebSocket.** WebRTC is better for browser clients (built-in audio handling). WebSocket is better for server-side relay (you control everything but have to handle audio yourself).
- **Latency is the product.** Anything that introduces a pause — slow tools, slow network, large models — kills the experience. Use fast tools, fast models, edge proximity.
- **Conversation state lives in the session.** Don't try to manage history yourself; the Realtime API handles it.
- **Handle interruption gracefully.** Users will talk over the model. The Realtime API has primitives for this — use them.
- **Cost is per-minute of audio in/out.** Track it. Long calls add up.

Use cases:

- Phone-based AI agents (customer support, scheduling).
- Voice assistants embedded in apps.
- Real-time translation.
- Accessible voice UIs.

Misuses:

- Non-real-time workflows (use STT → LLM → TTS sequentially; cheaper and simpler).
- Cases where text is fine (don't add voice for the demo wow).
- High-stakes irreversible actions without approval gates.

### Combining workflows

The most interesting products combine these patterns. A few examples:

- **Voice + tool use + RAG**: a phone-based support agent that looks up customer info during the call.
- **Deep research + scheduled tasks**: a daily competitive intel briefing delivered each morning.
- **Computer use + reasoning**: an agent that operates a legacy app to complete a complex workflow, using a reasoning model to plan the steps.
- **Voice + deep research**: ask a research question by voice, get a written report.

The pattern is always: primitives composed into a workflow, with guardrails at every step.

### Reliability in multi-step systems

The more steps, the more places things break. Patterns that help:

- **Idempotency.** Tool calls should be safe to retry. Use idempotency keys for state-changing operations.
- **Checkpoints.** Save state after each major step so failures don't lose work.
- **Timeouts.** Every step gets a budget. Exceeding it triggers fallback or abort.
- **Step-level retries with backoff.** Transient failures should retry; deterministic failures should escalate.
- **Logs at every step.** Multi-step debugging without logs is impossible.
- **Stop conditions.** Always have a hard cap on iterations. An agent in a loop is expensive and dangerous.
- **Human-in-the-loop fallback.** If confidence drops or iterations max out, route to a human.

These aren't unique to AI — they're standard distributed-systems patterns. But AI systems amplify the consequences of skipping them: a regular system fails loudly; an AI system fails *creatively*.

### Observability for multi-step systems

You need to see what each step did, in order, with inputs and outputs. Practical patterns:

- **Trace IDs.** Every user request gets one. Every step within the request inherits it. Logs are filterable by trace.
- **Per-step latency.** P50 / P95 per step, not just overall. Tells you where to optimize.
- **Per-step cost.** Multi-step pipelines blow budgets silently. Track which step is the expensive one.
- **Eval per step.** Don't just eval the final output. Eval the planning step, the search step, the synthesis step.

A multi-step agent that's hard to debug becomes impossible to maintain. Invest in observability before you scale.

## An analogy: a movie production

A single LLM call is a snapshot. A multi-step workflow is a movie production.

- The **director** (reasoning model or orchestrator) decides what to shoot in what order.
- The **camera operators** (tools) execute the specific tasks.
- The **editor** (final summarization step) cuts everything into a coherent story.
- The **producer** (your code) handles budget, schedule, and stops shooting when it's a wrap.

Single-call AI is improv. Multi-step AI is filmmaking. Both can be excellent, but the discipline is different. Improv is about reaction speed; filmmaking is about coordination, planning, and quality control.

The teams that ship great multi-step AI products treat them like productions: structured, observable, with clear handoffs between steps.

## Three real-world scenarios

**Scenario 1: The deep research feature that actually shipped.**
A B2B SaaS team built a feature that gave sales reps a one-paragraph briefing on any prospect — sourced from web search, news, and the company site. They tried it with regular GPT-5 and got shallow summaries reps ignored. They switched to a deep-research model with citations. Reps loved it. Per-query cost: $0.80. Reps used it 3x per call, saving 15 minutes of prep. Net: huge value.

**Scenario 2: The computer-use agent that worked in demo, failed in prod.**
A team built a computer-use agent to file expense reports in a legacy app. In the demo, it worked perfectly. In production, the app had a dozen edge cases — popup ads, A/B-tested layouts, sudden modal dialogs — that confused the agent. After 3 months of refinement, it handled 70% of cases reliably. They added a "fallback to human" route for the other 30%. Net positive, but humbling.

**Scenario 3: The voice agent that nailed first response, struggled with follow-ups.**
A team built a phone-based scheduling agent. The first turn ("book me an appointment") worked great. Subsequent turns ("actually make it Tuesday instead") sometimes confused the model — it would book a second appointment instead of modifying. They added explicit tool definitions for `modify_appointment` vs `create_appointment` and tightened the system prompt. Follow-up accuracy jumped from 60% to 92%. Lesson: tool design is half the work in voice agents.

## Common mistakes to avoid

- **Deep research used for fast queries.** It's slow and expensive; pick the right pattern.
- **Computer use against live production systems without sandboxing.** Recipe for disaster.
- **Voice agents with slow tools in the hot path.** Latency is the product. Slow tools break the experience.
- **No observability across steps.** Multi-step bugs are unfixable without logs.
- **No stop conditions on agent loops.** Runaway agents are expensive and dangerous.
- **Treating it all as 'just prompts'.** Multi-step systems need the same engineering rigor as any distributed system.

## Summary

- **Deep research**: long-running, citation-rich answers built on web search + reasoning. Minutes per query, dollars per query. Worth it for analytical depth.
- **Computer use**: screenshot-based agent that operates UIs. Powerful for legacy apps; needs sandboxing, approval gates, and careful eval.
- **Voice agents**: built on the Realtime API. Sub-second latency, bidirectional streaming, tool use mid-call. Latency is the product.
- **Combine workflows** for the most differentiated products: voice + tools, research + scheduling, etc.
- Multi-step systems need **idempotency, checkpoints, timeouts, retries, stop conditions, and human-in-the-loop fallbacks.**
- **Observability per step** is non-negotiable. Trace IDs, per-step latency, per-step cost, per-step evals.
- Treat multi-step AI as a distributed system, not as "prompt engineering."
